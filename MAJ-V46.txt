> Workflow Superpowers complet (systematic-debugging → writing-plans →

CAUSE RACINE IDENTIFIÉE
───────────────────────
Trois flux acompte coexistent dans le projet :
  1. **Encaissement (admin)** — `PopupEncaisserAcompte.tsx`
  2. **Déclaration (espace-client)** — `PopupAcompte.tsx`
  3. **Affichage KPI** — `AcomptesEncaisser`, `DetailDevis`, generators PDF

Le helper `validerNouveauPaiement` (`quoteStatusHelpers.ts:98-114`) déléguait
à `getSoldeRestant` (ligne 63) qui filtre par `encaisse === true` en interne.

Conséquence en chemin DÉCLARATION (`PopupAcompte.tsx:96` qui passe TOUS les
acomptes) : le `restant` retourné ne déduisait QUE les acomptes encaissés,
pas les déclarations en attente. L'admin/client pouvait donc empiler N
déclarations dont la somme totale dépasse le total devis, sans aucun
garde-fou en amont du flow d'encaissement.

De plus :
  ▸ `prochainPaiementEstSolde` comptait uniquement les encaissés
  ▸ `peutAjouterAcompte` idem

→ La règle business "max 3 acomptes" était **contournable** tant qu'aucune
déclaration n'était encaissée.

SCÉNARIO REPRODUCTIBLE (cas Michel V46)
───────────────────────────────────────
Devis 25 000 €, 1 acompte 500 € encaissé.
restant calculé par `validerNouveauPaiement` = 25 000 − 500 (encaissé) = 24 500 €

  Déclaration 1 — 600 €  → 600 ≤ 24 500 ✓ (mais devrait être 600 ≤ 24 500 − futurs)
  Déclaration 2 — 700 €  → 700 ≤ 24 500 ✓ (24 500 inchangé, le 600 ignoré)
  Déclaration 3 — 23 200 €  → 23 200 ≤ 24 500 ✓ (toujours 24 500)

Total déclarations + encaissé = 25 000 €. **4 acomptes** déclarés (violation
règle "max 3"). État incohérent affiché à l'admin.

DÉCOUVERTE ANNEXE
─────────────────
`devisHelpers.ts:52` — `getMontantRestantAVerser` calcule déjà correctement
le restant (encaissés + déclarés en attente) avec un commentaire explicite
ligne 47 : *« Différent de getSoldeRestant() qui ne soustrait que les
encaissés. »*. Cette fonction existe mais n'était pas utilisée par les
validateurs. Quelqu'un avait identifié le pattern conceptuel sans pousser
le fix dans la chaîne de validation.

FIX APPLIQUÉ
────────────
src/lib/quoteStatusHelpers.ts :

(1) Ajout helper `getNbAcomptesDeclares(acomptes)` — count tous non-soldés
    (encaissés + déclarés non-encaissés).

(2) `peutAjouterAcompte` utilise désormais `getNbAcomptesDeclares < 3` au
    lieu de `getNbAcomptesEncaisses < 3` → max 3 sur les déclarations
    totales.

(3) `prochainPaiementEstSolde` idem → la règle "4ème = solde forcé" se
    déclenche dès qu'on a 3 déclarations (peu importe encaisse).

(4) `validerNouveauPaiement` calcul du restant remplacé :
    AVANT : `restant = getSoldeRestant(total_ht, acomptes)` (filtre encaisse=true)
    APRÈS : `restant = total_ht - sum(acomptes.filter(!is_solde).montant)`
    → utilise TOUS les acomptes passés (peu importe encaisse).

    Le caller décide ce qu'il passe :
      ▸ chemin DÉCLARATION : passe `acomptes` complet → restant strict ✓
      ▸ chemin ENCAISSEMENT : passe `acomptes.filter(a.encaisse===true)` →
        restant = total_ht − encaissés (sémantique inchangée)

(5) Message d'erreur conforme spec V46 :
    «Acompte impossible : reste à payer = X €, vous saisissez Y €»

NON modifiés (sémantique correcte préservée pour l'affichage KPI réel) :
  `getTotalEncaisse`, `getSoldeRestant`, `getNbAcomptesEncaisses`
