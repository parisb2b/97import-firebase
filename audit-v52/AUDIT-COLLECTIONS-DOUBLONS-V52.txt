═══════════════════════════════════════════════════════════════
  4. ACOMPTES — ARRAY vs SUB-COLLECTION
═══════════════════════════════════════════════════════════════

  Schema actuel : array d'acomptes embedded dans le document
                  quote (champ `acomptes: [{ montant, date,
                  type, ... }]` + `total_encaisse` cache).

  Avantages array :
    - 1 read = quote + tous ses acomptes
    - Atomicite native (transaction Firestore)
    - PopupEncaisserAcompte / quoteStatusHelpers exploitent
      directement raw.acomptes
    - Numerotation acomptes simple (index 0, 1, 2)

  Inconvenients array :
    - Limite 1 MB par document (theorique, mais en pratique
      max 3 acomptes par devis donc impossible a saturer)
    - Pas de query cross-devis sur acomptes (dashboard CA
      doit iterer tous les devis)

  Verdict : PAS de doublon constate.
            Aucune sub-collection acomptes detectee en src/.
            Le pattern array est volontaire et stable depuis
            V44-V46.

  Recommandation : MAINTENIR le schema array. Migration
                   sub-collection serait un projet majeur (V60+)
                   et ne resout aucun probleme actuel.

  Action V52 : CP H va creer un script READ-ONLY
               (audit-acomptes-v52.cjs) qui parcourt quotes,
               compte les acomptes, valide la coherence
               total_encaisse vs sum(acomptes[].montant) et
               liste les desync. Le script ne sera PAS execute
               en V52. Michel le lancera apres V52 en --dry-run.
