  BUG 1 — SAUVEGARDE ALÉATOIRE FICHE PRODUIT  ✅ RÉSOLU
═══════════════════════════════════════════════════════════════════════════

> Priorité HAUTE — investigation cause racine demandée et obtenue.

CAUSE RACINE IDENTIFIÉE
───────────────────────
Stale-state cross-component. La fiche produit possède DEUX flux d'écriture
INDÉPENDANTS qui touchent le même document Firestore mais ne partagent pas
leur state local :

  Flux A — FicheProduit.handleSave() (bouton « Enregistrer le produit »)
    ▸ src/admin/pages/FicheProduit.tsx:135-148 (avant fix)
    ▸ payload = { ...product, completude, champs_incomplets, updated_at }
    ▸ setDoc(ref, payload, { merge: !isCreation })
    ▸ Spread TOUS les champs du state parent (≈40 champs incluant prix_*).

  Flux B — OngletGestionPrix.handleValidatePrices()
        et OngletGestionPrix.handleSaveOverrides()
    ▸ src/admin/services/pricingService.ts:216-249 (validateProductPrices)
    ▸ src/admin/components/produit/OngletGestionPrix.tsx:222-239 (overrides)
    ▸ Écrit prix_achat_cny/usd/eur, prix_public, prix_partenaire,
      prix_*_override, date_derniere_validation directement en BD.
    ▸ Le state parent FicheProduit.product n'est JAMAIS resynchronisé.

Scénario reproductible (déterministe, pas de race OS) :
  1. Admin ouvre fiche  → state parent product.prix_achat_cny = 100 (initial load)
  2. Rubrique 1 → MODE ÉDITION → CNY = 200 → VALIDER PRIX
       → Firestore = 200, state parent = 100 (jamais re-fetché)
  3. Onglet Essentiel → modif nom_fr → ENREGISTRER LE PRODUIT
       → setDoc({ ...product, ... }, merge:true) → écrit prix_achat_cny = 100
       → Firestore = 100 (le 200 validé en étape 2 est ÉCRASÉ)
  4. Reload post-save (FicheProduit.tsx:175-178) recharge depuis Firestore
       → écran montre 100. L'admin voit son prix sauter et conclut « save random ».

SCORING DES 5 HYPOTHÈSES UTILISATEUR
────────────────────────────────────
  1. Race setDoc vs calculerCompletude        ❌ calculerCompletude est sync
  2. undefined casse sanitizeForFirestore     ❌ handleSave ne l'utilisait pas ;
                                                  un undefined aurait throw
                                                  → toast erreur (pas silencieux)
                                                  Renforcé en defense-in-depth.
  3. useEffect rebascule la valeur saisie     ❌ useEffect FicheProduit:47 mount-only,
                                                  useEffect OngletEssentiel:83 creation-only
  4. State local pas synchronisé avec draft   ✅ Bonne intuition mais granularité
                                                  fine : la désync est entre
                                                  FicheProduit.product et les
                                                  écritures externes de
                                                  OngletGestionPrix
  5. Bouton submit pendant un save en cours   ❌ Bouton disabled={saving}

FIX APPLIQUÉ
────────────
Whitelist d'exclusion dans handleSave : on retire du payload les champs gérés
exclusivement par OngletGestionPrix, puis grâce à merge:true, ces champs
restent intacts en Firestore lors du Save « principal ».

  - Champs exclus : prix_achat_cny, prix_achat_usd, prix_achat_eur, prix_achat
                    prix_public, prix_partenaire,
                    prix_public_override, prix_partenaire_override,
                    date_derniere_validation,
                    id (le doc ID ne doit pas être stocké comme un champ)
  - Defense-in-depth : payload passé par sanitizeForFirestore avant setDoc
                       (élimine tout undefined futur).

  BUG 3 — « INVALID DATE » RUBRIQUE 1 GESTION DES PRIX  ✅ RÉSOLU
═══════════════════════════════════════════════════════════════════════════

CAUSE
─────
src/admin/components/produit/OngletGestionPrix.tsx:250 et :281 affichaient :
  Card subtitle = `Dernière validation : ${formatDateHeure(product.date_derniere_validation)}`

Pour un produit jamais validé, formatDateHeure retournait '—' → UX peu claire
(« Dernière validation : — » lu comme une date manquante / Invalid Date).
De plus, après VALIDER PRIX, la date n'apparaissait qu'après round-trip
Firestore (le state parent FicheProduit.product n'étant pas resynchronisé,
il fallait recharger la page — même problématique sous-jacente que Bug 1).

FIX APPLIQUÉ
────────────
1. State local optimiste localValidationDate dans OngletGestionPrix.
2. Memo subtitleValidation :
     - si localValidationDate ?? product.date_derniere_validation présent
         → "Dernière validation : <JJ-M-AAAA HHhMM>" (formatDateHeure)
     - sinon → "Jamais validé"
3. handleValidatePrices : après succès, setLocalValidationDate(new Date())
   → affichage immédiat sans attendre Firestore.
4. useEffect : reset du local quand la prop product transmet la vraie
   Timestamp Firestore (la prop devient source de vérité).

Cohérent avec la logique V44-BIS Bug 2 (Taux RMB optimistic update).

   FicheProduit.product reflète la BD en temps réel. Hors scope V44-TER.

   reload), ce bouton revient à la valeur AVANT validation. Bizarre mais
   rare en pratique. Voir observation 4 pour la solution propre.

   (handleSaveOverrides) mais le code de validateProductPrices côté
   pricingService.ts l'utilise aussi. Cohérence OK.
