# V50-BIS — Plan formel d'initialisation BD Firestore

> **For agentic workers:** Ce plan documente CP F + CP G de la mission V50-BIS.
> Aucune exécution Firestore depuis Claude Code. Les scripts sont commit-ready
> mais Michel exécute manuellement post-mission (rule V50-BIS).
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** initialiser proprement les collections Firestore manquantes
(`counters/{8 codes}`, `clients/{uid}`) tout en préservant les données
existantes (`users/{uid}`) et en couvrant tous les edge cases (ULTRATHINK).

**Architecture:** 2 scripts Node Admin SDK avec garde-fous projet hardcoded
+ exécution `--dry-run` obligatoire avant `--execute`. Pas de Cloud Functions
(pas d'infra), pas de Firestore Rules touchées (déjà déployées V49 RBAC).

**Tech Stack:** Firebase Admin SDK v13.x (firebase-admin npm), Node 20+,
firebase-admin-sdk.json (gitignored, présent à la racine).

| `FM` | Factures marchandises | FM-AAMM-NNN |
| `SAV` | Tickets SAV | SAV-AAMM-NNN |

Document Firestore initial (idempotent) :
```js
counters/{code} = {
  count: 0,
  prefix: <code>,
  last_reset: serverTimestamp()
}
```

### Garde-fou projet (CRITIQUE V50-BIS)
```js
const PROJECT_ID = 'importok-6ef77';
if (serviceAccount.project_id !== PROJECT_ID) {
  console.error('❌ ABORT : SDK cible ' + serviceAccount.project_id);
  process.exit(1);
}
```

### Mode `--dry-run`
- Liste les counters absents et ce qui SERAIT créé
- Aucune écriture
- `process.exit(0)` à la fin

### Mode normal (default)
- Vérifie chaque counter
- Si déjà existant → skip + log valeur courante
- Si absent → set avec `{ count: 0, prefix, last_reset: serverTimestamp() }`
- `process.exit(0)`

### Edge cases pris en charge
- Counter déjà existant avec count > 0 → NE PAS RESET (préserve historique)
- Counter avec format incorrect (champ `value` au lieu de `count`) → log warning, ne pas écraser
- BD inaccessible → process.exit(1) avec message clair
- service account absent → process.exit(1)
- service account project_id mismatch → process.exit(1)

## CP G — Migration users → clients (ULTRATHINK exhaustif)

### Stratégie
- `users/` reste source de vérité auth (Firebase Auth ne se déplace pas)
- `clients/{uid}` = vue enrichie pour le métier (préférences, adresses,
  historique commande, partenaire associé)
- doc id = UID Firebase (alignement avec V49 RBAC `match /clients/{userId}`)
- Migration une seule fois (champ `migrated_from_users_at` track)
- Idempotent : re-run skip les déjà-migrés

### Source = `users/{uid}` filtré sur :
- `role !== 'admin'` (admins ne sont pas des clients)
- `role !== 'partner'` (partenaires séparés dans `/partners/`)
- inclus : `role === 'client'` OU `role === null/undefined` (legacy)

### Champs clonés (best-effort, fallback null)
```js
{
  uid: <doc id>,
  email: data.email || null,
  nom: data.nom || data.lastName || '',
  prenom: data.prenom || data.firstName || '',
  telephone: data.telephone || data.phone || null,
  partenaire_code: data.partenaire_code || null,
  pays_livraison: data.pays_livraison || 'MQ',
  adresse_livraison: data.adresse_livraison || {},
  adresse_facturation: data.adresse_facturation || {},
  tva_intracom: data.tva_intracom || null,
  notes_admin: '',
  created_at: data.created_at || serverTimestamp(),
  migrated_from_users_at: serverTimestamp(),
  migrated_from_users_v: 'V50-BIS',
}
```

### Edge cases ULTRATHINK

#### EC-1 — User sans email
**Situation :** `users/{uid}` créé via Firebase Auth provider (Google,
téléphone) où email peut être absent.
**Solution :** logger + email = null. PAS bloquant. Le doc client est
créé sans email.

#### EC-2 — Email dupliqué entre plusieurs UID
**Situation :** Firebase Auth permet rarement les doublons mais possible
avec providers multiples ou imports manuels.
**Solution :** chaque UID a son propre doc client (UID-keyed). Pas de
conflit. Si Michel veut dédupliquer après, c'est un script séparé V51+.

#### EC-3 — User déjà dans `clients/`
**Situation :** Re-run du script ou migration partielle antérieure.
**Solution :** `existing.exists` → log skip. Idempotent.

#### EC-4 — Champs avec casing variant (camelCase vs snake_case)
**Situation :** `users/{uid}` peut avoir `firstName` (anglais) ou
`prenom` (français) selon ancienneté du compte.
**Solution :** fallback chain `data.prenom || data.firstName || ''`.
Compatible avec les 2 conventions.

#### EC-5 — User créé avec custom_claims uniquement (pas de doc users)
**Situation :** Firebase Auth user sans doc `users/{uid}` correspondant.
**Solution :** `db.collection('users').get()` n'inclura PAS ces users.
Donc ne sont pas migrés vers `clients/`. Acceptable pour V50-BIS — peut
être traité en V51 via `admin.auth().listUsers()` séparé.

#### EC-6 — Service account du mauvais projet
**Situation :** firebase-admin-sdk.json téléchargé d'un autre projet.
**Solution :** garde-fou `serviceAccount.project_id !== PROJECT_ID` → abort.

#### EC-7 — Quota Firestore atteint pendant migration
**Situation :** trop de users (>1000) → quota daily writes free tier.
**Solution :** N=7 users dans le projet actuel (nb mentionné par Michel).
Pas un risque immédiat. Pour future-proof : ajouter `--limit N` arg.
**Décision V50-BIS :** pas implémenté (YAGNI). Si problème, ajouter en V51.

#### EC-8 — Document client malformé après migration
**Situation :** données sources corrompues (objet array au lieu de map, etc.)
**Solution :** try/catch par doc dans la boucle. Log error + continue.
Ne pas arrêter toute la migration sur 1 doc cassé.

#### EC-9 — User créé pendant la migration (race)
**Situation :** un nouveau user s'inscrit pendant le run du script.
**Solution :** acceptable — ce user sera traité au prochain run du script
(idempotent). Pas critique pour migration one-shot.

#### EC-10 — Service account avec permissions insuffisantes
**Situation :** SA admin n'a pas Firestore write permission.
**Solution :** chaque écriture try/catch → erreur loggée, count `errors`.
process.exit(errors > 0 ? 1 : 0). Michel voit clairement la liste.

#### EC-11 — Données sensibles RGPD
**Situation :** RGPD permission de copier les données vers une nouvelle
collection.
**Solution :** `clients/` est dans le même projet Firebase, même
juridiction. Pas de transfert hors UE. RGPD OK (même base légale que
`users/`). **Note :** documenter l'opération dans le rapport MAJ-V50-BIS.txt
pour traçabilité.

### Mode `--dry-run` (OBLIGATOIRE en premier)
- Lit tous les `users/`
- Pour chaque candidat → log "serait créé" + preview des champs
- Compteurs : migrated, skipped, errors
- ZÉRO write
- `process.exit(0)` (sauf erreur lecture)

### Mode `--execute`
- Mêmes étapes mais set effectif
- Stop le script avec `process.exit(errors > 0 ? 1 : 0)` (Michel voit échecs)

### Vérification post-migration (à la main)
```bash
# Console Firebase
# /clients/ → doit avoir N docs avec migrated_from_users_at
# Tester /admin/clients → liste affiche les clients
```

## Risques et points de rollback

### Risque R1 — Migration partielle
- Script crash en milieu d'exécution
- Mitigation : idempotent (re-run skip déjà-migrés)
- Rollback : aucun (pas de mutation users/, juste clones)

### Risque R2 — Couverture incomplète
- Certains users non comptabilisés (custom_claims sans doc users)
- Mitigation : EC-5 documenté ; V51 traitera via listUsers() Auth
- Rollback : N/A

### Risque R3 — Données corrompues
- Champs sources malformés produisent des docs clients étranges
- Mitigation : try/catch par doc + report `errors` count
- Rollback : `db.collection('clients').doc(uid).delete()` manuel

### Risque R4 — Mauvais projet ciblé
- Service account d'un autre projet
- Mitigation : garde-fou `PROJECT_ID === serviceAccount.project_id`
- Rollback : abort avant write

### Risque R5 — Conflit avec V49 RBAC rules
- V49 a déjà déployé les rules pour `/clients/{userId}` 
- Action : aucun conflit puisque création de docs respecte les rules
  (admin SDK bypass les rules pour tout admin)
- Validation : tester `/admin/clients` après migration
